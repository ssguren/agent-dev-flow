# /draft-api-spec

基于需求文档和 ADR，生成 REST API 接口规范草稿。

---

## 用法

```
/draft-api-spec
/draft-api-spec <feature-name>
```

不带参数时，Agent 会扫描 `docs/requirements/` 下尚未有对应接口规范的需求文档，列出候选后由用户选择。

---

## Step 1：读取上下文文件

按以下顺序读取，任一文件不存在时记录缺口并继续：

1. **需求文档**：`docs/requirements/<feature>.md`（或用户指定路径）
   - 提取：功能描述、用户操作列表、业务规则、权限要求
2. **ADR**：`docs/adr/` 下所有文件
   - 提取：与该功能相关的架构决策（认证方案、分页约定、错误码规范等）
3. **现有接口规范**：`docs/api-spec/` 下所有已有规范
   - 提取：现有的路径前缀、命名惯例、响应格式约定
4. **项目 CLAUDE.md**（项目根）
   - 提取：技术栈、响应格式、状态码约定

读取完成后，输出"上下文摘要"：
```
已读取：
- 需求文档：<路径>（<行数>行）
- ADR：<N> 份，关联项：<列表>
- 现有规范：<N> 份，路径前缀：<前缀>
缺口：<未找到的文件列表>
```

---

## Step 2：从需求推导接口

### 推导规则

**用户操作 → 接口映射**

| 用户操作类型 | HTTP 方法 | 说明 |
|---|---|---|
| 查看列表 | GET | 通常需要分页、过滤、排序参数 |
| 查看详情 | GET | 路径参数携带 ID |
| 创建 | POST | 请求体携带数据，返回创建结果 |
| 编辑/更新全量 | PUT | 路径参数携带 ID + 请求体 |
| 部分更新 | PATCH | 仅更新指定字段 |
| 删除 | DELETE | 路径参数携带 ID，考虑软删除 |
| 状态变更（审批/发布等） | POST | 独立动作接口，如 `POST /articles/{id}/publish` |
| 批量操作 | POST | 请求体为 ID 列表，如 `POST /articles/batch-delete` |

**识别 CRUD 模式**

扫描需求中的动词和名词，识别主要实体，自动推导标准 CRUD 接口集合。若存在非标准操作（如"审批"、"归档"、"导出"），单独列出。

**路径命名约定**（优先使用项目 CLAUDE.md 中的约定）

- 复数名词：`/articles`，`/user-profiles`
- 使用 kebab-case
- 嵌套资源：`/orders/{orderId}/items`（嵌套不超过 2 层）
- 动作接口：`/resource/{id}/action-name`

## Step 3：输出格式

每个接口按以下结构输出：

```markdown
### <接口名称>

**Method:** `<HTTP_METHOD> <path>`

**描述：** <一句话说明该接口做什么>

**请求头：**
| 字段 | 必填 | 说明 |
|---|---|---|
| Authorization | 是 | Bearer <token>，需登录接口必填 |
| Content-Type | 是/否 | 有请求体时必填，值为 `application/json` |
| X-Tenant-Id | 视项目 | 多租户场景必填 |

**路径参数：**（如有）
| 参数名 | 类型 | 说明 |
|---|---|---|
| {id} | Long | 资源唯一标识 |

**查询参数：**（GET 接口，如有）
| 参数名 | 类型 | 必填 | 默认值 | 说明 |
|---|---|---|---|---|
| page | Integer | 否 | 1 | 页码，从 1 开始 |
| pageSize | Integer | 否 | 20 | 每页数量，最大 100 |
| sort | String | 否 | createdAt,desc | 排序字段和方向 |

**请求体：**（POST/PUT/PATCH，如有）
| 字段名 | 类型 | 必填 | 说明 |
|---|---|---|---|
| title | String | 是 | 标题，长度 1-200 |
| content | String | 否 | 正文内容 |
| status | String | 否 | 枚举值：DRAFT / PUBLISHED |

**响应体（成功）：**
\`\`\`json
{
  "code": 0,
  "traceId": "abc123",
  "data": {
    "id": 1001,
    "title": "示例标题",
    "createdAt": "2025-01-01T00:00:00Z"
  }
}
\`\`\`

**响应体（错误）：**
| 错误场景 | code | message |
|---|---|---|
| 参数校验失败 | 40001 | 字段 xxx 不能为空 |
| 资源不存在 | 40401 | 资源不存在 |
| 无权限 | 40301 | 无操作权限 |
| 并发冲突 | 40901 | 数据已被修改，请刷新后重试 |

**HTTP 状态码：**
| 场景 | 状态码 |
|---|---|
| 成功 | 200 |
| 创建成功 | 200（统一 200，code=0 表示成功） |
| 服务器错误 | 200（code 非 0） |

**权限要求：**
- 登录用户：是/否
- 角色限制：<角色名>（如无限制则填"无"）
- 数据范围：仅本人 / 本租户 / 全局
```

### 列表接口额外输出

列表接口的 `data` 字段需包含分页信息：

```json
{
  "code": 0,
  "data": {
    "list": [...],
    "total": 100,
    "page": 1,
    "pageSize": 20,
    "totalPages": 5
  }
}
```

## Step 4：输出"需要人工确认的设计决策"清单

完成接口草稿后，**必须**输出以下格式的确认清单。不要自行拍板这些问题。

```markdown
## 需要人工确认的设计决策

以下问题需要 Tech Lead / Product 确认后，才能完成接口规范：

- [ ] **[权限]** <接口名> 的数据范围是"仅本人"还是"同租户所有人"？
- [ ] **[分页]** 列表接口是否需要游标分页（适合大数据量）还是页码分页？
- [ ] **[字段]** <字段名> 的枚举值有哪些？是否需要多语言 label？
- [ ] **[删除]** 删除操作是物理删除还是软删除（`deletedAt` 字段）？
- [ ] **[版本]** 并发更新场景是否需要乐观锁（`version` 字段）？
- [ ] **[文件]** <接口> 是否涉及文件上传？如有，文件大小和类型限制是什么？
- [ ] **[异步]** <操作> 是同步返回结果，还是异步任务（返回 taskId 轮询）？
```

只列出本次草稿中确实存在不确定性的问题，不要生成与本次需求无关的通用问题。

---

## 输出位置

生成的规范草稿保存到：`docs/api-spec/<feature-name>.md`

若该文件已存在，输出差异部分并提示用户确认是否覆盖。
