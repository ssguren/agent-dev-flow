# Command: review-api-spec

**角色**：Frontend Dev Agent
**用途**：从前端视角审查后端接口规范，确保接口满足 UI 实现需求。输出 MUST-FIX 和 NICE-TO-HAVE 问题清单，供 Backend 和 Tech Lead 决策。Agent 不直接修改接口规范。

**调用方式**：
```
/review-api-spec
/review-api-spec <spec-file-path>
/review-api-spec <feature-name>
```

**参数说明**：
- 不带参数：审查 `specs/api-specification.md` 或 `docs/api-spec/` 下所有未审查的接口规范
- 带路径时：审查指定文件
- 带功能名时：匹配 `docs/api-spec/<feature-name>.md` 或包含该功能名的规范文件

---

## Step 1：读取文档

按以下顺序读取，记录不存在的文件：

1. **接口规范文档**
   路径：`specs/api-specification.md` 或 `docs/api-spec/<feature>.md`（或用户指定路径）
   提取：所有 endpoint、请求/响应字段类型、状态码、枚举值、分页参数。

2. **前端设计稿/交互原型**
   路径：`docs/design/<feature>.md` 或 `docs/design/` 下相关文件，或文档中注明的 Figma 链接
   提取：页面结构、用户操作流程（列表/详情/表单等 UI 模式）、实时更新需求、文件上传场景。

3. **需求文档**
   路径：`docs/requirements/<feature>.md`
   提取：用户故事和验收条件中涉及 UI 的部分，特别是搜索、筛选、排序、批量操作的需求描述。

4. **项目 CLAUDE.md**（项目根）
   提取：前端框架、已有的分页方案、日期格式约定、国际化要求。

读取完成后输出：
```
已读取：
- 接口规范：<路径>，共 <N> 个 endpoint
- 设计稿：<路径 或 "未找到，仅基于需求文档审查">
- 需求文档：<路径 或 "未找到">
- 项目 CLAUDE.md：<已读取 或 "未找到，使用默认前端约定">
```

---

## Step 2：逐维度系统性检查

对每个维度，检查所有相关 endpoint，汇总发现的问题。

### 维度 1：列表接口的分页、排序、过滤能力

前端列表页通常需要分页、排序、过滤/搜索。检查每个 GET 列表接口：

| 检查项 | 问题描述 | 严重程度 |
|--------|---------|---------|
| 是否支持分页 | 无分页参数（`page`/`pageSize` 或游标）时，数据量大会导致性能问题，前端无法控制展示范围 | MUST-FIX |
| 响应是否含分页元数据 | 无 `total`/`totalPages` 时，前端无法显示"第 X 页，共 Y 页" | MUST-FIX |
| 是否支持排序 | 列表页若有排序 UI，接口无 `sort` 参数时前端只能在客户端做假数据排序（数据量大时不可行） | MUST-FIX |
| 是否支持过滤/搜索 | 搜索框/筛选器所需的查询参数是否都有对应的接口参数 | MUST-FIX |
| 是否存在无限滚动需求 | 若 UI 为无限滚动，页码分页不适合，需改为游标分页（`cursor` 或 `lastId`） | NICE-TO-HAVE |

### 维度 2：时间字段格式

检查所有时间类型字段（`createdAt`、`updatedAt`、`startTime`、`endTime` 等）：

| 检查项 | 问题描述 | 严重程度 |
|--------|---------|---------|
| 格式是否明确 | 未注明是 Unix timestamp（毫秒/秒）还是 ISO 8601 字符串 | MUST-FIX |
| 时区是否明确 | ISO 8601 是否含时区（`Z` 或 `+08:00`），timestamp 是否说明是秒还是毫秒 | MUST-FIX |
| 前端格式化成本 | 若返回秒级 timestamp，前端需 `× 1000`，容易出错且难排查；建议统一毫秒或 ISO 8601 | NICE-TO-HAVE |

### 维度 3：枚举值的可读性

检查所有枚举类型字段（`status`、`type`、`role` 等）：

| 检查项 | 问题描述 | 严重程度 |
|--------|---------|---------|
| 枚举值列表是否完整 | 规范中未列出所有可能的枚举值，前端无法做完整的 switch 处理，可能有未处理情况 | MUST-FIX |
| 是否有多语言需求 | 若产品支持国际化，枚举 label 应由后端返回或通过字典接口提供，不能前端硬编码 | MUST-FIX（如有 i18n） |
| 是否只有 code | 接口只返回 `"DRAFT"`、`"PUBLISHED"` 等 code，前端需自己维护中文映射表，枚举值增加时容易漏更新 | NICE-TO-HAVE |

### 维度 4：响应数据结构的嵌套层级

检查所有响应体的 JSON 结构：

| 检查项 | 问题描述 | 严重程度 |
|--------|---------|---------|
| 是否存在循环引用风险 | 嵌套结构中是否存在 A 包含 B、B 又包含 A 的情况，会导致 JSON 序列化崩溃 | MUST-FIX |
| 嵌套超过 3 层 | `data.user.profile.address.city` 这类深层嵌套增加前端解包成本和空值判断负担（可选链写到崩） | NICE-TO-HAVE |
| 列表接口是否内嵌完整关联对象 | 列表接口若每项都内嵌完整的关联对象（如完整用户信息），会导致响应体过大，建议只返回 ID + 必要字段 | NICE-TO-HAVE |

### 维度 5：批量操作接口

检查是否存在需要批量操作的 UI 场景（列表勾选、批量发布、批量删除等）：

| 检查项 | 问题描述 | 严重程度 |
|--------|---------|---------|
| 列表勾选后批量操作 | 若无批量接口，前端需要发 N 个串行或并行请求，产生 N+1 问题，体验差且后端压力大 | MUST-FIX |
| 批量状态变更 | 如批量发布、批量归档等操作是否有对应接口 | MUST-FIX |
| 批量请求的大小限制 | 接口是否说明单次批量的最大数量，前端需在 UI 上限制或分批处理 | NICE-TO-HAVE |

### 维度 6：文件上传接口

检查涉及文件上传的接口：

| 检查项 | 问题描述 | 严重程度 |
|--------|---------|---------|
| Content-Type 是否明确 | `multipart/form-data` 还是 Base64 字符串，影响前端实现方式完全不同 | MUST-FIX |
| 文件大小限制 | 未说明最大文件大小，前端无法做客户端校验，只能等后端报错 | MUST-FIX |
| 文件类型限制 | 接受哪些 MIME 类型或扩展名，前端需在 `<input accept="">` 中配置 | MUST-FIX |
| 上传结果 | 返回文件 URL 还是文件 ID，影响后续如何引用该文件 | MUST-FIX |
| 上传进度 | 是否需要显示上传进度条（需要流式处理，普通 JSON 接口无法支持） | NICE-TO-HAVE |

### 维度 7：实时更新场景（WebSocket / SSE）

检查是否存在需要实时更新的 UI 场景（消息推送、状态变更通知、进度展示等）：

| 检查项 | 问题描述 | 严重程度 |
|--------|---------|---------|
| 实时场景是否有对应接口 | 若 UI 有实时更新需求（如聊天、通知、进度条），但只有 REST 接口，前端只能轮询（体验差且耗费资源） | MUST-FIX |
| 消息格式是否规范 | 消息体的 JSON 结构是否有文档，是否有消息类型字段用于路由不同类型的事件 | MUST-FIX |
| 鉴权方式 | WebSocket 握手时如何传递 token（header 还是 query 参数，两者实现方式不同） | MUST-FIX |
| 重连机制 | 连接断开时如何重连，是否需要前端实现指数退避，后端是否保留消息队列 | MUST-FIX |
| 心跳设计 | 是否有 ping/pong 心跳，超时时间是多少，影响前端保活策略 | NICE-TO-HAVE |

### 维度 8：错误信息的前端可用性

检查所有接口的错误响应：

| 检查项 | 问题描述 | 严重程度 |
|--------|---------|---------|
| 错误信息是否对用户友好 | `message` 字段是否可以直接在 UI 上展示给用户，还是需要前端维护 code-to-label 映射表 | MUST-FIX |
| 表单校验错误的字段级信息 | 表单提交失败时，错误信息是否细化到字段级（`errors.email = "格式不正确"`），还是只有一条全局错误 | MUST-FIX |
| 错误 code 列表是否完整 | 规范中是否枚举了所有可能的错误 code，前端无法处理未文档化的错误 code | NICE-TO-HAVE |

## Step 3：输出审查结果

审查结果输出到对话窗口，不修改任何接口规范文档。

输出格式如下：

```markdown
## API 规范前端审查报告

**审查功能**：<feature-name>
**规范文件**：<path>
**审查时间**：<date>
**接口数量**：<N> 个

---

### MUST-FIX（必须修改，否则前端功能无法实现或实现代价极高）

#### [M1] <问题标题>

**接口**：`<METHOD> <path>`
**问题**：<描述问题是什么>
**影响**：<说明不修改会导致什么 UI 问题>
**建议方案**：<提出修改建议，仅供参考，最终由 Backend 和 Tech Lead 决策>

#### [M2] ...

---

### NICE-TO-HAVE（建议改进，当前可绕过，但会增加前端复杂度或影响用户体验）

#### [N1] <问题标题>

**接口**：`<METHOD> <path>`
**问题**：<描述问题是什么>
**影响**：<说明不修改的代价，以及可能的前端绕过方案>
**建议方案**：<提出修改建议>

#### [N2] ...

---

### 无问题接口

以下接口通过前端审查，无需修改：
- `GET /api/v1/<path>`
- ...

---

### 文档缺失

以下文件未找到，审查存在盲区：
- （列出 Step 1 中未找到的文件，说明可能遗漏的检查维度）
```

输出结束后，打印以下提示：

```
以上问题请提交给 Backend Dev 和 Tech Lead 评审。
确认修改范围后，再使用 /task-start 开始前端实现。
```

---

## 重要约束

- 不直接修改接口规范文档
- 建议方案仅供参考，最终修改方案由 Backend 和 Tech Lead 决策
- 审查结果不自动触发任何后续操作，所有后续步骤由人工决定
