# /gen-consensus-doc

基于产品资料生成需求评审共识文档（阶段一A）。

共识文档的目标读者是全员（PM、后端、前端、测试），所有内容使用业务语言，非技术人员可以理解。这是进入详细设计前的必经步骤。

---

## 使用方式

```
/gen-consensus-doc
```

执行前请确认已提供产品资料（PRD 文档 / Figma MCP / 蓝湖 MCP / GitHub 文档仓库链接）。如未提供，先询问资料来源模式。

---

## 执行步骤

### 第一步：确认资料来源模式

根据当前提供的资料，判断属于哪种模式：

| 模式 | 资料组合 |
|------|----------|
| A — PRD + Figma | PRD 文档 + Figma 设计稿（MCP 获取） |
| B — PRD + 蓝湖 | PRD 文档 + 蓝湖设计稿（MCP 获取） |
| C — GitHub 双仓库 | 文档仓库（需求）+ Lovable 代码仓库（设计原型）+ Issue（补充） |

列出已获取的资料清单，指出缺失或不明确的部分。如果 PRD 与设计稿存在矛盾，明确指出冲突点，不自行决定以哪个为准。

### 第二步：生成共识文档

按以下结构输出，存入 `docs/consensus/<feature>.md`：

---

```markdown
# <功能名称> — 共识文档

> 版本：YYYY-MM-DD | 状态：待评审

## 资料清单

- [ ] PRD / 需求文档：<来源说明>
- [ ] 设计稿：<来源说明，或"无设计稿，基于需求文字还原">
- [ ] 已知 Issue / Gap：<来源说明，或"无">

缺失或不明确的部分：
- <列出，若无则写"资料完整">

---

## 业务流程图

> 判断标准：以下场景必须生成流程图：
> 1. 涉及状态流转（审批流、订单状态、支付流程）
> 2. 涉及多角色交互（买家-卖家-平台）
> 3. 涉及条件分支（不同用户类型走不同路径）
> 4. 涉及异步处理或回调（支付回调、消息通知）
>
> 简单 CRUD 操作不需要流程图。

```mermaid
flowchart TD
    ...
```

> 流程图必须体现：正常流程、异常分支（失败/超时/拒绝等）、状态终态。
> 节点使用业务语言命名（如"用户提交申请"，不是"POST /api/apply"）。

> 以下问题不影响当前版本功能范围，但会影响底层设计。请产品以"是 / 否 / 暂不确定"回答：

### 第三步：输出完成提示

共识文档生成后，输出：

```
共识文档已生成：docs/consensus/<feature>.md

下一步建议：
1. 组织需求评审会，全员（PM、后端、前端、测试）共同确认
2. 产品填写"业务前瞻性确认"表格中的 5 个问题
3. 评审通过后，分别开新对话：
   - 后端：/gen-detailed-design-be（Backend Agent）
   - 前端：/gen-detailed-design-fe（Frontend Agent）
   - 测试：/gen-test-plan（QA Agent）
```

## 语言规范（强制）

共识文档中**禁止出现**：
- 技术术语：DDD、Gateway、Domain、Service 编排、分层架构、Repository、Controller
- 框架名：Spring Boot、React、Taro、sxp-framework、antd（可提"后端服务"、"前端页面"）
- 代码元素：包名、类名、注解名、配置项
- 数据库术语：Collection、Index、Schema、外键（用"记录"、"信息"代替）

共识文档中**应该使用**：
- 业务操作：用户提交、系统通知、商家查看
- 业务实体：订单、模板、用户、任务
- 业务结果：成功/失败、生效/过期、已完成/待处理

---

## 不属于此命令的输出

以下内容**不在共识文档范围内**，留给阶段一B 各角色处理：
- 后端分层实现（Gateway 接口定义、Domain 业务规则、Service 编排逻辑）
- 前端组件拆分方案、状态管理方案
- 具体测试用例
- 数据库索引优化策略、慢查询预防
- DDD 模块划分、聚合根等领域设计术语
