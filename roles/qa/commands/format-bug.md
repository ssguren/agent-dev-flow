# /format-bug — 将 Bug 描述格式化为标准 Bug Report

## 使用说明

**方式一：先执行命令，再粘贴描述**
```
/format-bug
```
执行后粘贴 Bug 描述，Agent 从描述中提取已有信息，列出缺失字段后等待补充。

**方式二：命令 + 描述一次性输入**
```
/format-bug
---
登录页面点击登录按钮后没有跳转，停在登录页，但后台日志显示登录成功了。
测试环境，staging，使用的是 MEMBER 角色账号。
---
```

---

## Step 1：从描述中提取已有信息

从用户输入中提取以下字段，**能提取多少算多少，不猜测，不补全**：

| 字段 | 提取方式 |
|---|---|
| 标题 | 从描述中归纳一句话标题，不超过 80 字符，格式：`[模块] 功能描述 + 异常表现` |
| 严重级别 | 根据关键词初步判断（见下方判断依据），必须标注"（初步判断，需确认）" |
| 复现步骤 | 从描述中提取操作序列，整理为有序步骤列表 |
| 预期结果 | 从描述中提取，或根据功能名称推断（标注"（推断）"） |
| 实际结果 | 从描述中直接提取，尽量精确 |
| 测试环境 | 从描述中提取（dev / staging / production） |
| 账号角色 | 从描述中提取 |
| 是否稳定复现 | 从描述中提取（必现 / 概率复现 / 仅一次） |
| 涉及模块 | 从接口规范或代码结构推断（标注"（推断）"） |

**严重级别初步判断依据：**

| 级别 | 判断条件 | 典型场景 |
|---|---|---|
| P0 | 系统不可用 / 数据丢失 / 安全漏洞 | 生产崩溃、用户数据泄露、支付金额错误 |
| P1 | 核心功能不可用，无绕过方案 | 登录不通、订单无法提交、文件无法上传 |
| P2 | 功能异常但有绕过方案 | 搜索结果排序错误、导出格式不对 |
| P3 | 体验问题或细节错误 | 文案错别字、图标偏移、提示语不准确 |

---

## Step 2：识别缺失字段，列出需要补充的内容

对以下字段，如果无法从描述中提取，列出问题等待补充，**不要猜测**：

**必填字段（缺少则不输出 Bug Report，改为列出缺失清单）：**
- 复现步骤（至少 1 步操作）
- 预期结果
- 实际结果

**重要字段（缺少则在报告中标注 `[待补充]`）：**
- 测试环境（dev / staging / production）
- 账号角色（用什么角色账号触发）
- 浏览器 / 客户端版本
- 是否可稳定复现（必现 / X% 概率 / 仅出现一次）
- 截图或日志（标注"需要附件"）

**可选字段（能推断就填，标注来源；不能就留空，不猜测）：**
- 涉及接口（从接口规范推断，标注"（推断）"）
- 涉及模块（从代码结构推断，标注"（推断）"）
- 出现版本 / 正常版本

---

## Step 3：推断可能涉及的模块

从以下来源推断 Bug 可能涉及的模块，**推断结果必须标注"（推断）"并说明推断依据**：

1. **接口规范**：操作涉及哪个接口，该接口属于哪个模块（需说明：从哪个接口推断到哪个模块）
2. **项目目录结构**：根据功能名称推断代码路径（需说明：根据什么功能名推断到什么路径）
3. **错误信息中的关键词**：日志中的类名、包名、错误码（需说明：从哪条日志信息推断）

不能推断的情况（如描述中完全没有模块相关信息）：留空，标注"无法从描述推断，请确认"。

---

## Step 4：输出标准 Bug Report

```markdown
## Bug Report

**标题**：[模块] 功能描述 + 异常表现

**严重级别**：P1（初步判断，需确认）
**状态**：待确认

---

### 环境信息

- **测试环境**：staging
- **服务版本**：[待补充]
- **客户端**：Chrome 120 / iOS 17 / Postman（根据描述填写）
- **账号角色**：MEMBER
- **是否稳定复现**：必现 / [待补充]

---

### 复现步骤

1. 打开登录页面
2. 输入合法的用户名和密码
3. 点击"登录"按钮

---

### 预期结果

点击登录后，页面跳转至首页，登录态生效。

### 实际结果

点击登录后，页面停留在登录页，未发生跳转。后台日志显示登录接口返回成功（HTTP 200，code=0）。

---

### 补充信息

**涉及接口**（推断）：`POST /api/v1/auth/login`（推断依据：操作为登录，接口规范中登录接口为该路径）
**涉及模块**（推断）：认证模块 / 前端路由模块（推断依据：后端返回成功，问题可能在前端跳转逻辑）
**相关日志**：[待补充]
**截图**：[待补充]

---

### 备注

[其他需要说明的上下文，或空]
```

---

## 不该做的事

- **不给 Bug 定根因**：不写"是因为 SQL 写错了"、"应该是没加事务"——根因分析是开发的工作，Bug Report 只描述现象
- **不猜测复现步骤**：用户没说清楚如何操作时，列出问题等待补充，不凭空补全步骤
- **不判断"是否是 Bug"**：即使看起来像需求理解问题，也如实记录让团队确认，不在 Bug Report 中下结论
- **不省略"（推断）"标注**：所有非用户直接提供的信息，必须标注来源或标注"（推断）"
- **不过度推断严重级别**：初步判断的级别必须标注"（初步判断，需确认）"，最终由 QA 负责人或产品确认
