# Agent Dev Flow — 项目背景

## 我们在解决什么问题

软件开发是一个多角色协作的过程。一个功能从"想法"到"上线"，至少要经过产品、设计、后端、前端、QA、运维六个角色的接力。

**现实中，每一次接力都在漏信息、漏责任、漏上下文。**

---

## 全流程的七个摩擦点

```
需求 → 设计 → 开发 → Review → 测试 → 部署 → 监控
  ↑      ↑      ↑       ↑       ↑      ↑      ↑
  [1]    [2]    [3]     [4]     [5]    [6]    [7]
```

**[1] 需求 → 设计**
PM 用产品语言写需求，技术侧用技术语言做设计。
没有系统性的机制把"产品意图"翻译成"技术约束"，也没有把"技术风险"反馈给"产品决策"。
结果：设计方案与需求意图之间产生 Gap，到开发阶段才被发现，返工成本高。

**[2] 设计 → 开发**
架构决策停留在口头或飞书，没有结构化归档。
新人上手、跨团队协作、追溯历史决策时，没有可靠的单一来源。
结果：同样的架构问题被反复讨论，"为什么这样设计"无法回答。

**[3] 开发阶段内部**
每个开发者各自和自己的 Agent 交互，上下文不共享。
同一个项目里，A 的 Agent 做了某个约定，B 的 Agent 不知道。
结果：代码风格割裂，接口约定不一致，注释语言不统一。

**[4] Review**
代码 Review 依赖个人经验和时间。
Reviewer 看不到需求上下文，只能就代码论代码；
作者解释不清楚"为什么这样写"，因为背景分散在飞书和口头沟通里。
结果：Review 要么流于形式（快速通过），要么陷入争论（缺上下文）。

**[5] 测试**
测试计划由 QA 基于对需求的理解独立编写，与后端实现和接口规范脱节。
边界情况、错误路径、权限矩阵往往覆盖不全。
结果：测试完了上线，线上出现的 Bug 是测试阶段没覆盖到的场景。

**[6] 部署**
上线前的检查清单靠经验，靠人记，靠运气。
配置项、数据迁移、外部依赖的就绪状态没有系统性验证。
结果：上线翻车的大多数原因不是代码 Bug，而是"忘了改某个配置"。

**[7] 监控 / 复盘**
线上问题修完就结束，根因分析没有归档，下次遇到同类问题重头来。
没有机制把线上问题的根因反馈回需求和设计层。
结果：同类问题反复出现，每次都是"救火"，没有"防火"。

---

## 根本原因

上面七个摩擦点，表象各不同，根因只有三个：

| 根因 | 具体表现 |
|------|---------|
| **翻译成本高** | 每个角色用自己的语言思考，没有中间层把不同语言对齐 |
| **上下文断裂** | 每个阶段的输出不是下一个阶段的输入，信息在接力中丢失 |
| **决策无归档** | 决策停在对话里，无法检索，无法追溯，无法传递 |

---

## Agent 能做什么，不能做什么

Agent 擅长：
- **翻译**：把产品语言翻译成技术问题，把技术问题翻译成产品影响
- **生成**：基于上下文生成分析、草稿、检查清单、测试用例
- **检查**：对照规范、对照历史决策做一致性检验
- **归档**：把口头决策、对话结论结构化写入文档

Agent 不能做：
- **决策**：决策需要人来负责，Agent 只能提供选项和分析
- **跨系统感知**：Agent 不知道上下文以外发生了什么
- **替代沟通**：人与人之间的信任和理解无法被 Agent 代替

**因此，这个项目的核心设计原则是：**

> **Agent 负责生成和分析，人负责确认和决策。**
> **Agent 降低每次接力的翻译成本，人保证每次接力的上下文完整传递。**

---

## 这个项目要做什么

`agent-dev-flow` 是一套**开发全流程的 Agent 协作规范和工具集**，覆盖从需求到上线的完整链路。

它不是一个框架，不是一个平台，而是一套**约定 + 命令 + 模板**：

- **约定**：每个阶段的输入是什么、输出是什么、谁负责确认
- **命令**：每个角色在每个阶段可以调用的 Agent 命令（slash commands）
- **模板**：每个阶段的标准化文档结构，保证上下文能完整传递到下一阶段

目标：**让每一次接力都有上下文，让每一个决策都有归档，让每一个角色都有 Agent 协助，但每一个责任都由人承担。**

---

## 设计思考

这一节记录在构建这套框架时做出的关键设计判断，以及每个判断背后的取舍逻辑。写下来的目的不是自我解释，而是让后续维护者能理解"为什么是这样设计，而不是别的样子"，从而做出更好的修改。

---

### 一、为什么从 PM-BE 摩擦出发，而不是从代码质量出发

这个框架的第一个版本不是凭空设计的，而是从一个具体的 PM-BE 协作问题中生长出来的：产品写了需求，技术侧发现了 Gap，但两边都没有系统性的机制处理这些 Gap。

之所以选择从这里入手，而不是从代码质量、测试覆盖率这些更常见的切入点出发，是因为**上游的摩擦比下游的漏洞成本更高**。一个需求理解错误导致的返工，比一个 Bug 要贵得多；一个被搁置的 Gap 最终变成上线后的争议，比一个测试没覆盖的边界情况要难处理得多。

大多数工程质量工具（CI、Lint、Code Review 规范）解决的是下游问题。这个框架想解决的是更上游的：**信息在角色之间流动时为什么会丢失，以及如何系统性地减少这种丢失**。

---

### 二、为什么是「文档流」而不是「工具链」

设计这套框架时面临的第一个选择：是建一个平台（Jira / Confluence / 内部系统），还是围绕 Git 文档做约定？

选择文档流的理由：

**工具链有锁定成本，文档没有。** 一套 Jira 流程绑定了团队的工具栈。当团队换工具、扩大规模或换人时，流程往往也跟着断。但 Markdown 文档在 Git 里，任何工具都能读，任何 Agent 都能理解。

**Agent 天然理解文档，不天然理解平台 API。** 让 Claude Code 读一个 `gap-registry.md` 然后生成分析，比让它调用 Jira API 拉取工单列表要可靠得多，也更通用。只要约定了文档结构，Agent 就能接入流程，不需要专门的集成。

**文档即上下文。** 这是整个框架最核心的设计假设：每个阶段的输出文档，就是下一个阶段 Agent 的输入上下文。这个假设只在文档是结构化的、在版本控制下的、可被引用的情况下成立。工具链的数据往往散落在 SaaS 系统里，很难作为 Agent 上下文传递。

代价是：**文档流依赖团队的文档纪律**。如果没人更新 gap-registry，这套流程就失效了。这是有意接受的取舍——一个需要工具执行才能维持的流程，本质上是在用工具弥补纪律缺失；而一个依赖文档纪律的流程，意味着团队真的理解并认同这套约定的价值。

---

### 三、为什么 `confirmed` 状态只有人能触发

Gap Registry 的状态机中，`be_reviewed` 和 `pm_reviewed` 可以由 Agent 修改，但 `confirmed` 和 `closed` 只有人能修改。这不是技术限制，是刻意的设计。

核心考量：**决策必须有人负责，才有执行的可能**。

如果 Agent 能把一个 Gap 标记为 `confirmed`，那这条 Gap 的决策就没有责任人。下次有人问"为什么选了这个方案"，答案是"Agent 说的"——这不可追溯，也无法在产品发生变化时重新评估。

更深的问题是：**Agent 分析的质量依赖它所掌握的上下文**，而上下文总是不完整的。Agent 不知道本季度的商务谈判进展，不知道 PM 和投资人上周开了什么会，不知道团队的真实交付压力。人知道这些，但很难把这些都写进文档。因此，人工确认不只是走流程——它是把那些"说不清楚但真实存在"的上下文注入到决策里的唯一方式。

这个设计的代价是：**流程有摩擦**。每条 Gap 必须等人来 confirm，不能全自动流转。这是有意为之的摩擦——它迫使人真正读完 Agent 的分析，而不是签字盖章。

---

### 四、为什么是七个阶段，而不是五个或十个

最初的草稿把"监控"单独列为第七阶段，后来改成了"复盘（Postmortem）"。这个改动背后的判断是：**监控是基础设施，复盘才是流程节点**。

监控（Grafana 看板、告警规则、日志系统）是一个独立的技术体系，和开发流程的关系更多是"工具"而不是"阶段"。但复盘是一个角色参与、有输入有输出、有责任人的流程节点——它应该在这个框架里。

七个阶段的划分依据是**角色切换**，不是时间线：

- 需求阶段：PM 主导
- 设计阶段：Tech Lead 主导
- 开发阶段：Dev 主导
- Review 阶段：非 Author 的人主导（视角切换）
- 测试阶段：QA 主导
- 部署阶段：DevOps 主导
- 复盘阶段：全员参与

每次主导角色切换，就是一次接力，就是一次上下文可能丢失的风险点。七个阶段对应七个风险点，不多不少。

---

### 五、为什么复盘阶段要强制反馈到上游

复盘（Stage 7）的设计中有一条强制规则：**每次复盘必须产生至少一条对上游文档的修改**。这个规则容易被忽略，但它是整个框架能否形成闭环的关键。

没有反馈闭环的流程是一次性的：每次都从零开始积累经验，然后下次换一批人，经验又归零。有了反馈闭环，每次复盘的结论都沉淀在文档里，变成下一次 Sprint 的起点。

具体反馈路径：
- 发现需求描述不清晰 → 更新需求文档模板
- 某类 Bug 反复出现 → 更新 CLAUDE.md 编码规范
- 部署失败因为漏了步骤 → 更新部署清单模板
- Agent 生成质量差 → 更新对应命令文件的提示词

这些更新看起来很小，但它们是知识从"个人经验"转化为"团队规范"的唯一路径。

---

### 六、关于命令设计的取舍

当前框架只实现了最高价值的 6 个命令（`/review-pr`、`/draft-adr`、`/gen-test-plan`、`/gen-deploy-checklist`、`/write-pr-desc`、`/gen-test-plan`）。还有十几个命令在 `workflow/` 文档中被引用但尚未实现。

这是有意为之的决定，而不是未完成的工作。

过早实现所有命令有两个风险：
1. **命令过多导致没人用**。一个框架有 20 个命令，每个命令做 5 件事，最终的结果是大家谁都记不住，需要的时候不知道用哪个，最终放弃整套框架。
2. **命令的真正需求在实际使用中才会清晰**。`/split-tasks` 该怎么切，取决于这个团队的任务粒度习惯；`/gen-sprint-retro` 需要哪些数据，取决于这个团队实际产出哪些文档。先用已有的命令，在真实使用中发现"这一步很痛，如果有个命令就好了"，再去写，命令的质量会更高。

因此：**已有命令是可以立即使用的，缺失的命令是需要在使用中发现需求后再填的**。

---

### 七、这套框架本身的最大风险

诚实地说，这套框架有一个内生的最大风险：**文档纪律无法被系统强制，只能被文化维持**。

如果团队里有人不更新 gap-registry，如果 PM 的需求文档不遵循模板，如果 Dev 在提 PR 时没有跑 `/write-pr-desc`，整套流程就会在最脆弱的地方断掉。

这个风险没有技术解法。真正的解法是：**团队需要共同经历一次"因为上下文断裂导致的严重返工"，并且认同这套框架能防止下次再发生**。

框架只有在被理解为"对自己有好处的工具"而不是"被要求遵守的流程"时，才会被自发地维护。这是任何协作规范都面临的根本问题，不只是这个框架特有的。

---

## 适用场景

- 小型技术团队（3-15 人），每个人都使用 Claude Code 或类似 Agent 工具
- 有明确的 Sprint 节奏，需求来源清晰（PM 写文档 / Issue）
- 使用 Git 作为协作基础，文档和代码都在版本控制下
