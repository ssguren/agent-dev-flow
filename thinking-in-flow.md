# Thinking in Flow — agent-dev-flow 的前世今生

> 这不是一篇使用文档，也不是架构设计说明。这是一篇回顾：这个项目从哪里来，遇到了什么，做了哪些判断，以及它正在走向哪里。
>
> 写给未来的维护者，也写给任何想理解"为什么是这样"而不只是"是什么"的人。

---

## 一、起点不是想法，是一次真实的摩擦

这个项目不是凭空设计的。

它的起点是 2026 年 3 月的一个具体场景：一个后端团队在开发过程中，PM 写了需求文档，工程师发现了技术上的问题和分歧，但两边都没有结构化的机制处理这些分歧。问题存在于飞书、存在于口头会议、存在于每个人的理解里——就是没有存在于任何一个两边都能看到的地方。

更具体地说：工程师用 Claude Code，PM 用 Claude Web。两个人都在用 AI 辅助工作，但他们的 AI 彼此不知道对方做了什么。每次需要对齐，都是一次从零开始的信息同步。

这种摩擦不是 PM 或工程师的个人问题，而是协作结构上的问题。没有共享的上下文层，每次跨角色交接都在漏信息。

**agent-dev-flow 最初就是为了解决这一个问题。**

---

## 二、第一个选择：文档，而不是平台

面对"信息漏失"的问题，最直觉的解法是"上一套系统"：Jira、Notion、飞书文档、内部 Wiki……这些工具都在某种程度上试图解决同样的问题。

我们没有选这条路。原因很简单：

**工具有锁定成本，文档没有。**

一套 Jira 流程绑定的不只是数据，还有操作习惯、权限设置、通知配置。换人、换团队、换工具，流程就断了。但 Markdown 文件在 Git 里——任何工具都能读，任何 Agent 都能理解，任何人都能从命令行操作。

更关键的是：**Claude Code 天然理解文档，不天然理解平台 API**。让 Agent 读一个 `gap-registry.md` 然后生成分析，远比让它调用 Jira API 拉工单列表可靠。只要约定了文档结构，Agent 就能接入流程，不需要专门的集成。

这个选择带来了一个代价，也是整个框架最大的内生风险：**文档纪律无法被系统强制，只能被文化维持**。如果没有人更新 gap-registry，流程就在最脆弱的地方断掉。我们知道这个风险，有意接受了它。

---

## 三、从一个 Gap，到整条流水线

解决了 PM-BE 的 Gap 问题之后，一个更大的问题浮现了：

Gap 只是 Review 阶段之前的一个摩擦点。整个开发流程有七个交接：需求、设计、开发、Review、测试、部署、复盘。每一个交接都在漏信息，只不过方式不同。

于是框架的范围扩大了。从 Gap Registry 出发，逐渐覆盖了：
- 架构决策的归档（ADR）
- 接口规范的生成和校验
- PR 的上下文摘要和规范检查
- 测试用例的系统性生成
- 部署清单的自动化组装
- 复盘结论向上游的反馈闭环

每一个扩展都来自一个具体问题，不是预先设计好的。这也是为什么有些命令非常完善（`review-pr`、`gen-deploy-checklist`），有些只有骨架（`split-tasks`、`gen-sprint-retro`）——**命令的完成度反映的是这个痛点被真实遭遇过多少次**。

## 四、一条铁律是怎么形成的

在构建这套框架的过程中，有一个问题反复出现：

> Agent 能不能自己把 Gap 标记为 `confirmed`？

每次问这个问题，答案都是"不行"。但"不行"背后的理由值得说清楚：

**决策必须有人负责，才有执行的可能。**

如果一个 Gap 的 `confirmed` 状态是 Agent 打上的，那这条决策的责任人是谁？下次有人问"为什么选了这个方案"，答案是"Agent 说的"——这不可追溯，也无法在情况改变时被重新评估。

更深的问题是：Agent 的分析质量取决于它的上下文，但上下文总是不完整的。Agent 不知道本季度的商务压力，不知道 PM 上周和投资人开了什么会，不知道团队的真实交付节奏。这些东西很难被写进文档，但它们是真实存在的决策因素。

**人工确认不只是走流程，它是把那些"说不清楚但真实存在"的上下文注入到决策里的唯一方式。**

这条铁律的代价是：流程有摩擦。每条 Gap 必须等人来 confirm，无法全自动流转。这个摩擦是有意设计的——它迫使人真正读完 Agent 的分析，而不是签字盖章。

这个判断后来被写进了 `CLAUDE.md` 的铁律，贯穿整个框架。

## 五、遇到了一堵墙

框架搭好之后，遇到了第一个真实的使用问题：

**这套工具只有工程师能用。**

所有命令依赖 Claude Code CLI，而 PM、运营、QA 活在 Claude Web 里。要求她们切换到终端、理解 slash command，等于要求她们改变整个工作方式。这不现实。

但 Gap 流程的对侧是 PM，测试计划的对侧是 QA，部署确认的对侧是运维——如果这些角色不能直接参与，所谓"跨角色协作流程"其实是假的，只是让工程师在命令行里模拟了两个角色之间的对话。

这个问题没有"工程师教会 PM 用 CLI"的解法，因为问题不是技能，而是工作习惯的权重。

---

## 六、GitHub 作为共享上下文层

解法来自一个简单的观察：

> 如果 PM 不愿意来工程师的世界，那让工程师的工具走向 PM 的世界。

PM 熟悉 GitHub Issues（即使不熟悉，开个 Issue 的门槛也远低于用 CLI）。Claude Web 支持配置 MCP Server，而 GitHub 官方 MCP 覆盖了 Issues、PR、Comment 的完整读写。

这意味着可以设计一套新的协作结构：

```
PM（Claude Web + GitHub MCP）          工程师（Claude Code CLI）
        ↓                                         ↓
  创建 Issue / 回复 Comment                读 Issue / 写分析 Comment
        ↓                                         ↓
              GitHub 仓库（双方共享的单一上下文层）
```

两边用不同的工具，操作同一个仓库。任何一方的动作以 Issue / Comment / label 的形式落地，另一方下次读取时自然可见。

这不是一个新的协作平台，而是把现有的 GitHub 基础设施重新定义为"协作上下文的物理载体"。

关键约定只有一条：**对话是草稿区，GitHub 是决策区。** 停留在 Claude 对话里的内容对他人不可见；影响他人的决策必须通过 MCP 写入 GitHub，才算生效。

这和现实工作一样：你可以在本地草稿里改十遍，但只有发出去的那条消息才算数。

## 七、验证：一个 Issue 的完整生命周期

设计想法有了，但想法不等于能用。2026-03-25，做了一次端到端的测试。

测试仓库：一个专为测试搭建的协作私有仓，PM 和工程师各自是 Collaborator。

**TC-01**：PM 在 Claude Web 通过 MCP 创建了 Issue #1——`[GAP] V1 与 V2 登录方式需求冲突：邮箱密码登录 vs 谷歌登录`。Issue 打上了 `gap` + `high` label。

**TC-02**：工程师在 Claude Code CLI 运行 `/be-review-gaps`，通过 MCP 读取 Issue，生成技术分析（影响评估、方案对比、工期估算），写回 Comment，label 更新为 `be-reviewed`。

**TC-03**：PM 在 Claude Web 读取 BE 的技术分析，给出产品决策方向（以 V2 为准，接入 Google OAuth），写回 Comment，label 更新为 `pm-reviewed`。

**TC-04**：PM 手动将 label 改为 `confirmed`，关闭 Issue（state: completed）。整个流转当天完成。

测试暴露了一个真实问题（P1）：Docker 版 GitHub MCP 对协作者身份的私有仓读写失败，换成 `npx @modelcontextprotocol/server-github` 后解决。还有一个设计问题（P2）：`/be-review-gaps` 命令用 `raised` label 作为过滤条件，但实际流程中这个 label 不会被自动打上——命令需要修订，改为以 `gap` 为主过滤，排除已处理状态。

**这次测试证明了：用 GitHub Issues 作为双方共享上下文层是可行的。**

## 八、现在站在哪里

今天（2026-03-25），Claude Code 本身也可以使用 GitHub MCP 了。

这意味着工程师侧的命令也可以直接通过 MCP 读写 GitHub，而不只是依赖本地文件。`/review-pr` 可以直接读 PR 数据，而不是让人粘贴 diff；`/be-review-gaps` 可以主动拉取 Issues，而不是等人把内容复制进来；`/format-bug` 可以直接创建 GitHub Issue，而不是输出 Markdown 让人手动提交。

这条线，从"只有工程师能用的 CLI 工具"，走到了"PM 和工程师共享一套基础设施，各自用最顺手的工具操作"。

## 九、还没解决的问题

诚实地说，有些问题这个框架至今没有解法：

**自动化触发**。目前所有命令都是人主动触发的。当 PM 创建了 Issue，工程师不会自动收到通知说"你该跑 `/be-review-gaps` 了"。用 GitHub Actions 作为事件驱动层是下一步的目标，但还没有实现。

**文档纪律**。如果没有人更新接口规范、没有人维护 gap-registry、没有人跑复盘命令，这套流程就会在最脆弱的地方断掉。这个问题没有技术解法。它的真正解法是：团队需要共同经历一次"因为上下文断裂导致的严重返工"，然后认同这套框架能防止下次再发生。

**多项目复用**。这套命令目前需要手动复制到每个项目的 `.claude/commands/` 目录。一旦某个命令更新，已复制出去的版本不会自动同步。这个问题现在靠手动处理，需要更好的分发机制。

**Agent 能力边界**。所有命令的质量依赖输入文档的质量。如果接口规范写得模糊，`/gen-test-plan` 生成的测试用例也会模糊；如果 ADR 没有写，`/review-pr` 就没有上下文来评估架构合理性。框架提供结构，但填充结构的仍然是人。

## 十、设计的核心判断没有变

从最早的 Gap Registry，到七阶段全流程，再到 GitHub MCP 集成，表象在变，核心判断没有变：

**Agent 负责生成和分析，人负责确认和决策。**

这不只是一条铁律，它是整个框架的设计原点。一套让 Agent 替人做决策的工具，在表面上看起来更自动化、更高效——但它把责任模糊了，把追溯性消灭了，也把人从协作中移走了。

这个框架想要的不是让人从流程中消失，而是让人能更清晰地做自己该做的事：决策、确认、负责。

Agent 做的那部分是翻译、生成、检查、归档——这些事情人做起来慢且容易遗漏，Agent 做起来快且机械精确。但正因为是机械精确，才需要人来判断它的输出是否符合那些说不清楚、写不进文档的真实上下文。

**这是人机协作最合理的分工，也是这个框架存在的理由。**
